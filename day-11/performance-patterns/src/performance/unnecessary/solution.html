The Problem — Over-memoization

ProductCard.jsx

Both useMemo and useCallback are wrapping trivially cheap operations. They cost more than they save.

Before — unnecessary memoization 
function ProductCard({ price }) {
  // ❌ price * 0.18 is a single multiply —
  //    cheaper to run than useMemo's bookkeeping
  const tax = useMemo(
    () => price * 0.18,
    [price]
  );

  // ❌ arrow fn is recreated anyway in deps array
  //    and console.log is NOT expensive
  const logPrice = useCallback(
    () => console.log(price),
    [price]
  );

  return (
    <div>
      <h3>Price: {price}</h3>
      <button onClick={logPrice}>
        Log
      </button>
    </div>
  );
}
  • ❌ useMemo for a multiply: React has to allocate a closure, store deps array, and compare on every render — all for price * 0.18.
  • ❌ useCallback for console.log: The function body is trivial. React's overhead for tracking the callback exceeds the cost of recreating it.
  • ❌ Both depend on price: So they re-run every time price changes anyway — memoization provides zero benefit here.
  • ℹ️ Memoization is not free. It trades CPU (re-computation) for memory + comparison overhead. Only worthwhile when computation is genuinely expensive.
✅

The Fix — Plain variables & inline handlers

ProductCard.jsx

Compute directly. React 19 Compiler will automatically memoize the entire component output when props haven't changed.

After — clean, no manual memoization 
function ProductCard({ price }) {
  // ✅ Plain variable — runs inline, zero overhead
  const tax = price * 0.18;

  // ✅ Inline handler — React Compiler will
  //    memoize the whole render if price is stable
  return (
    <div>
      <h3>Price: {price}</h3>
      <p>Tax: {tax}</p>
      <button
        onClick={() => console.log(price)}
      >
        Log
      </button>
    </div>
  );
}
  • ✅ Simpler code: No hook imports needed for this component. Less cognitive overhead for future readers.
  • ✅ Same performance: price * 0.18 is a nanosecond operation — the JIT will inline it. No measurable difference.
  • ✅ React Compiler (Babel plugin) analyzes this component and automatically wraps the entire output in a stable memo boundary when price hasn't changed.
  • ℹ️ With React 19 Compiler enabled, you write plain React — the compiler decides what to memoize, not you.

⚙️ Why React Compiler Handles These Cases

React Compiler (previously "React Forget") is a static analysis tool that ships with React 19. It transforms your source code at build time — before it reaches the browser. It understands React's rules of hooks and component purity, so it can safely infer what needs to be memoized.

Manual useMemo/useCallback are hints to React at runtime. The compiler replaces that entire class of hints with compile-time guarantees — making your app faster without the cognitive burden.

🔍 Static Analysis

The compiler reads your component tree and builds a dependency graph. It knows price * 0.18 depends only on price — no side effects, no external refs.

🏷 Automatic Memo Boundaries

It wraps component outputs in optimized memo slots — equivalent to React.memo() but more granular. Subtrees that don't change are skipped entirely.

⚡ Stable Callbacks

Inline handlers like onClick={() => console.log(price)} are automatically stabilized when the compiler can prove their closure values haven't changed.

🛡 Safety Guarantees

The compiler only transforms code that follows React's rules (pure components, no direct mutation). Unsafe patterns are left untouched — no silent breakage.

When Memoization Is Still Required

manual override cases

The React Compiler cannot handle every case. You still reach for useMemo/useCallback manually in these scenarios:

Case Hook Why compiler can't handle it Status
Heavy computation
e.g. sorting 10k rows, matrix math, FFT		 useMemo Compiler memoizes render output, not arbitrary computations that run outside JSX. Manual required
Stable ref for external lib
e.g. passing callback to D3, socket.on(), observer		 useCallback External libraries hold references outside React's tree — compiler has no visibility there. Manual required
Custom hook that returns object/array
returned value used as useEffect dep		 useMemo Non-primitive return values change identity every render; compiler won't break that contract. Depends on hook
Context value object
e.g. <AuthContext.Provider value={{ user, logout }}>		 useMemo Context forces all consumers to re-render when value identity changes — compiler v1 doesn't optimize across Provider boundaries. Manual required
Opt-out of Compiler
legacy code, 3rd-party HOCs, non-pure components		 Both Compiler skips components that aren't provably pure. Manual memoization is the only option. Case-by-case
Pre-React 19 codebase
React 16–18 without compiler plugin		 Both Compiler is not available. Classic manual memoization is the only optimization tool. Still valid
useMemo still needed — heavy sort 
function LeaderBoard({ scores }) {
  // ✅ 10,000 items sorted on every render
  //    Compiler won't hoist this computation
  const sorted = useMemo(
    () => [...scores].sort(
      (a, b) => b.score - a.score
    ),
    [scores]
  );

  return <VirtualList items={sorted} />;
}
useCallback still needed — external ref 
function MapView({ onMove }) {
  // ✅ mapbox.on() holds a JS reference outside
  //    React's tree — must be stable identity
  const handleMove = useCallback(() => {
    onMove(map.getCenter());
  }, [onMove]);

  useEffect(() => {
    map.on('move', handleMove);
    return () => map.off('move', handleMove);
  }, [handleMove]);

  return <div id="map" />;
}
