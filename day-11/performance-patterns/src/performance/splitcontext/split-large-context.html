The Problem — One giant AppContext

AppContext.js

When all unrelated state sits inside one provider value object, any change to one field changes the entire object identity and re-renders all consumers.

Before — shared value object causes fan-out 
const AppContext = createContext({
  user: null,
  theme: "light",
  cart: [],
  locale: "en",
});

function AppProvider({ children }) {
  const [user, setUser] = useState(null);
  const [theme, setTheme] = useState("light");
  const [cart, setCart] = useState([]);
  const [locale, setLocale] = useState("en");

  const value = {
    user,
    theme,
    cart,
    locale,
    setUser,
    setTheme,
    setCart,
    setLocale,
  };

  return (
    <AppContext.Provider value={value}>
      {children}
    </AppContext.Provider>
  );
}
  • ❌ Cart changes rebuild the whole context value: even UserProfile and ThemeToggle re-render.
  • ❌ Unrelated consumers wake up: components reading only user still re-run when cart length changes.
  • ❌ Poor scaling: as more fields are added, every update fans out to more consumers.
  • ℹ️ Context compares provider value by identity, not by which property each consumer actually reads.
✅

The Fix — Split by update frequency and ownership

UserContext.js · ThemeContext.js · CartContext.js

Separate unrelated state into focused providers so updates stay local to the consumers that actually need them.

Contexts + providers 
export const UserContext = createContext(null);
export const ThemeContext = createContext("light");
export const CartContext = createContext([]);

function UserProvider({ children }) {
  const [user, setUser] = useState(null);
  return (
    <UserContext.Provider value={{ user, setUser }}>
      {children}
    </UserContext.Provider>
  );
}

function ThemeProvider({ children }) {
  const [theme, setTheme] = useState("light");
  return (
    <ThemeContext.Provider value={{ theme, setTheme }}>
      {children}
    </ThemeContext.Provider>
  );
}

function CartProvider({ children }) {
  const [cart, setCart] = useState([]);
  return (
    <CartContext.Provider value={{ cart, setCart }}>
      {children}
    </CartContext.Provider>
  );
}
App providers + updated consumers 
function AppProviders({ children }) {
  return (
    <UserProvider>
      <ThemeProvider>
        <CartProvider>
          {children}
        </CartProvider>
      </ThemeProvider>
    </UserProvider>
  );
}

function UserProfile() {
  const { user } = useContext(UserContext);
  console.log('UserProfile render');
  return <div>{user?.name}</div>;
}

function ThemeToggle() {
  const { theme, setTheme } = useContext(ThemeContext);
  console.log('ThemeToggle render');
  return <button>{theme}</button>;
}

function CartBadge() {
  const { cart } = useContext(CartContext);
  console.log('CartBadge render');
  return <span>{cart.length}</span>;
}
  • ✅ Cart update only touches cart consumers: UserProfile no longer re-renders on add-to-cart.
  • ✅ Cleaner ownership: user, theme, and cart now have separate provider lifecycles.
  • ℹ️ This is often a bigger win than adding React.memo everywhere, because it removes re-render triggers at the source.

function AddToCartButton() {
  const { setCart } = useContext(CartContext);

  return (
    <button
      onClick={() => {
        console.log('Add to cart click');
        setCart(prev => [...prev, { id: Date.now() }]);
      }}
    >
      Add to cart
    </button>
  );
}

Render Comparison

before vs after

Splitting context does not reduce the number of cart updates, but it sharply reduces how many unrelated components are forced to participate in each update.

Scenario UserProfile ThemeToggle CartBadge Result
Cart update with one AppContext Re-renders Re-renders Re-renders Broad fan-out
Cart update with split contexts Stays stable Stays stable Re-renders Scoped update
Theme update with split contexts Stays stable Re-renders Stays stable Targeted update
User login with split contexts Re-renders Stays stable Stays stable Targeted update
  • ✅ Best rule: split context by data that changes together, not by app page or folder.
  • ⚠️ If two values always update together, they may still belong in one context.
  • ℹ️ For even finer control, context selectors or external stores can go beyond this pattern — but splitting is the first clean fix.
